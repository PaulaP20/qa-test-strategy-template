# Definition of Done for QA

## Objective

Define the quality criteria that must be met before a feature or user story is considered complete from a QA perspective.

This helps ensure consistent quality standards across teams and reduces the risk of defects reaching production.

---

## QA Readiness Criteria

Before testing begins:

- Acceptance criteria are clearly defined
- Requirements are understood by QA
- Test environments are available and stable
- Test data is prepared if required

---

## Testing Requirements

A feature is considered tested when:

- Functional tests are executed
- Edge cases are validated
- Negative scenarios are covered
- Relevant regression tests are executed
- API and integration points are verified if applicable

---

## Defect Management

- All Blocker and Critical defects must be resolved
- Major defects must be evaluated before release
- Minor defects may be accepted depending on the impact

---

## Automation

When applicable:

- Automated tests are added or updated
- Existing regression automation is validated
- CI pipeline tests pass successfully

---

## Documentation

Before closing the work item:

- Test cases are updated
- Relevant documentation is updated if needed
- Known issues are documented

---

## Release Validation

Before production release:

- Smoke tests executed
- CI/CD pipeline successful
- Environment stability confirmed
- QA approval provided
